## What CU is

A generative AI product where you POST a JSON "analyzer" describing your
schema, then submit documents to its analyze endpoint. CU internally runs
OCR + a generative model (configured at the resource level) and returns
structured JSON conforming to the schema, with confidence scores and
grounding citations.

## Foundry deployment quota

Default GlobalStandard SKU on `gpt-5.2` is **capacity 10 = 10K TPM + 100 RPM**. Each CU analyze call sends ~5-8K tokens (image + 74-field schema + prompt + structured output), so capacity 10 fits ~1-2 calls/min. With 20 sample workflows in parallel (the backend's default concurrency), the long tail timed out the default 600 s sample timeout.

The fix is provider-specific capacity tuning: on the test resource we have 1M TPM regional quota free, so bumping the deployment to capacity 100 (= 100K TPM, ~15-20 calls/min) is one `az resource update --set sku.capacity=100` call ‚Äî no Microsoft support ticket required. The two operational knobs:

- **Deployment capacity** is the bottleneck for any single account. Bump it to whatever the regional quota allows. Pricing-wise capacity is a throughput cap, not a reservation ‚Äî you're billed per token used regardless.
- **Regional quota** caps the deployment ceiling. eastus2 limit is 1M TPM per subscription for `gpt-5.2-GlobalStandard`. If the CU workload needs > 1M TPM, request uplift via Microsoft support.

## Confidence-distribution observations

CU returns a per-field `confidence` ‚àà [0, 1] when
`config.estimateFieldSourceAndConfidence: true`. The mapper computes a
page-level confidence as the mean of per-field confidences (falls back to
0.95 when no confidences are present, matching the Mistral fallback so
the default 0.95 threshold in `ocr.checkConfidence` behaves consistently
across providers).

On the 40-sample canonical run, CU's per-field confidences cluster around
0.90-0.99 with a long tail down to ~0.4 for fields where the generative
pass was less certain (typically blank cells in income tables). The
page-level mean confidence sits in [0.85, 0.97] across samples, so the
default 0.95 threshold in `ocr.checkConfidence` *does* fire occasionally
on E03 ‚Äî unlike E02 where the mapper's canned 0.95 fallback meant HITL
never fired. This is a meaningful difference between the two providers:
E03 actually surfaces "I'm not sure" signals at the field level, whereas
E02 reports a constant.

## Retrospective ‚Äî what we learned setting up E03

This is a candid record of the things that surprised us, the workarounds we built, and what should change in the spec / `_shared-rules.md` and downstream experiments.

### Schema-level CU quirks that were not in any documentation we found at first

1. **`analyzerId` cannot contain `-`** ‚Äî `PUT /analyzers/myAnalyzer-x` returns HTTP 400 `InvalidAnalyzerId` / `"The 'analyzerId' cannot contain '-'"`. Microsoft Learn's prebuilt analyzers (e.g. `prebuilt-idDocument`) use `-`, but the rule applies only to custom analyzer IDs. We sanitise `${AZURE_CU_ANALYZER_PREFIX}-${templateModelId}` to lowercase alphanumeric (`diexperimentseedsdprmonthlyreporttemplate`) and never hit the rejection again. The doc's regex `^[a-zA-Z0-9.\_-]{1,64}$` shown for the URI param is misleading ‚Äî `-` is documented as allowed but rejected at runtime by the resource provider.

2. **`PATCH /contentunderstanding/defaults` must be called once before any PUT analyzer** ‚Äî without it every PUT analyzer fails with HTTP 400 `DefaultsNotSet`. CU expects three model aliases to be wired (`gpt-5.2` or `gpt-4.1`, `text-embedding-3-large`, plus the logical aliases `prebuilt-analyzer-completion` and `prebuilt-analyzer-embedding`). The Foundry resource we hit had no OpenAI deployments at all (only the Mistral deployment from E02), so we deployed `gpt-5.2` (GlobalStandard cap 10) and `text-embedding-3-large` (Standard cap 120) and wired both alias families.

3. **`gpt-4.1-mini` is *not* needed for custom analyzers** ‚Äî only for the prebuilt search analyzers (`prebuilt-*Search`). For schema-driven extraction with `baseAnalyzerId: prebuilt-document`, only the primary completion + embedding deployments are invoked. The Microsoft Learn quickstart deploys all three by default; we skipped the mini model and CU works fine.

4. **CU rejects `data:` URLs in `inputs[].url`** ‚Äî even though it accepts `inputs[].url` for public HTTPS URLs. The error is HTTP 400 `InvalidSource: "The request must contain a 'url' property."` (misleading message). The documented alternative for inline submission is `inputs[].data` (base64-encoded) + `inputs[].mimeType`. We use that path so we don't have to upload to a public URL first.

5. **PUT analyzer is not upsert** ‚Äî a second PUT with the same `analyzerId` returns HTTP 409 `ModelExists`. The PATCH endpoint exists but only updates `description` and `tags`, not `fieldSchema` / `models` / `config`. To change the schema we DELETE then PUT. The deploy activity probes via GET first and skips the round-trip when the deployed body matches the desired body (in-memory cache + remote-equality check).

6. **Newly-PUT analyzers have `status: creating`** ‚Äî POST `:analyze` against them returns HTTP 400 `ScenarioNotReady` until the status flips to `ready`. The deploy activity now polls the GET endpoint after PUT until status is `ready` (typically < 5 s).

7. **`gpt-5.4` and `gpt-5.5` exist in Azure but CU rejects them** ‚Äî CU's supported-completion list is currently `gpt-4.1` + `gpt-5.2` (per [Microsoft Learn](https://learn.microsoft.com/en-us/azure/ai-services/content-understanding/concepts/models-deployments)). Newer GPT-5.x models in the Azure catalog cannot be wired to CU's aliases yet. The doc explicitly says: *"GPT-5.2 is now supported across all Content Understanding analyzers. Support for additional models will be added in future updates."* The recommended model is `gpt-5.2` per the same page.

8. **Auth header** ‚Äî Microsoft Learn's quickstart uses `Ocp-Apim-Subscription-Key`, not `Authorization: Bearer`. Despite the brief's Foundry-Bearer assumption (carried from the Mistral provider), this route is documented as subscription-key-keyed and works that way out of the box. We default to subscription-key and expose `AZURE_CU_AUTH_MODE=bearer` for the Foundry-style override.

### Operational gotchas that cost us time

9. **The default `gpt-5.2` deployment capacity (10) caps throughput at 10K TPM**, which fits ~1-2 CU calls per minute. With 20 sample workflows in parallel (backend's default concurrency) the long tail blew past the default 600 s per-sample timeout, so the first run completed only 9 of 40 samples. The fix is a one-shot `az resource update --set sku.capacity=100` (within the 1M TPM regional quota) ‚Äî bumps throughput 10√ó and the canonical run completes 40 samples in ~15 min wallclock. We document the capacity bump as a pre-flight step in `_shared-rules.md`.

10. **The default `runtimeSettings.timeout` for `CreateRunDto` is 600 s** ‚Äî fine for E01/E02 where each sample finishes in 1-7 s, but too short for CU. Pass `runtimeSettingsOverride: { timeout: 3600, retries: 1 }` in the trigger body. The `scripts/run-experiment-benchmarks.sh` bash script doesn't expose this knob; we lifted the trigger into TypeScript (`apps/temporal/src/scripts/trigger-experiment-benchmark.ts`) so we can override per-experiment.

11. **`benchmark_ocr_cache` was empty after the canonical run** because the CU activity returned `{ ocrResult, cuResponse }` while the workflow's `outputs[port="ocrResponse"]` was looking for `ocrResponse`. Same naming pattern E02 codified. Renamed `cuResponse` ‚Üí `ocrResponse` in the activity return. Fix is on the branch; the canonical run's metrics are still valid (the cache emission failure is independent of the prediction quality).

12. **`TEST_API_KEY` lives in `apps/backend-services/.env`, not in `~/.config/bcgov-di/backend-services.env`** as the brief implied. The seed picks it up because backend-services' env-loader reads both files; our trigger script (running from the temporal app) had to extend its env-loading order to include `apps/backend-services/.env` to resolve `TEST_API_KEY` without leaking the value to the shell.

- **Idempotent deploy activity** ‚Äî `azureCuDeployAnalyzer` GETs first, compares the deployed body, and DELETE+PUTs only when changed. Polls until `status: ready` so the next analyze call doesn't hit `ScenarioNotReady`.

### What should change in `_shared-rules.md` and the spec

These rules were learned the hard way on E03 and should be in the canonical brief before E04 starts:

1. **CU defaults pre-flight** ‚Äî when the experiment uses CU, the first benchmark run requires `PATCH /contentunderstanding/defaults` with a deployment for each model alias. Without it, every PUT analyzer fails. Document the helper script as canonical.
2. **`analyzerId` charset** ‚Äî Microsoft's regex documents `^[a-zA-Z0-9.\_-]{1,64}$` but the resource provider rejects `-` for custom analyzers. Sanitise to lowercase alphanumeric.
3. **Capacity bump as a pre-flight** ‚Äî for CU and any future GPT-5.2-backed engine, `--sku-capacity 10` (= 10K TPM) is a default that does not fit 20-sample-parallel benchmarks. Bump to capacity 100 before running unless you know your token budget per call is < 0.5K. The 1M TPM regional quota means no support ticket needed for most cases.
4. **Per-sample timeout override** ‚Äî generative-engine experiments need `runtimeSettingsOverride: { timeout: 3600 }`. Update `scripts/run-experiment-benchmarks.sh` to accept a `SAMPLE_TIMEOUT_SECONDS` env var and pass it through, so future experiments don't have to fork the script.
5. **Sync-provider cache port name** ‚Äî the activity's return field MUST be `ocrResponse` (matching the workflow's outputs port). E02 codified this; the renaming in this branch documents that the convention is load-bearing, not stylistic.
6. **CU's PUT is not upsert** ‚Äî when an experiment iterates the analyzer body during prompt tuning, the deploy activity has to DELETE-then-PUT (or skip when bodies match). This is different from Mistral's behaviour ‚Äî annotation_format is sent inline per-request, not registered as a server-side resource.
7. **Use inline `data` + `mimeType`, not `data:` URLs** ‚Äî for any CU-derivative engine. The doc accepts the URL form for public URLs but rejects `data:` URLs.
8. **Note the supported-model list per page version** ‚Äî CU's supportedModels list updates over time. Future experiments should call GET `/analyzers/{analyzerId}` and read the `supportedModels` block before assuming a model alias is wired.

### Meta-process lessons (engine-independent ‚Äî apply to E04 + E05)

These are the things that wasted time on E03 *not* because of CU itself but
because of how the experiment was sequenced or how the harness/repo/scripts
behaved. Worth baking in before the next experiment starts.

1. **Pre-flight as a runnable script** ‚Äî E03 had four conditions that had
   to hold before the first iteration call (env vars set, model
   deployments exist, `PATCH /defaults` ran, deployment capacity ‚â• X).
   Three of them surfaced as runtime errors *during* iteration; each
   cost a re-run. Future experiments should ship a `preflight-<engine>.ts`
   that asserts every precondition upfront. E03 has a partial version at
   [`setup-cu-defaults.ts`](../../../apps/temporal/src/scripts/setup-cu-defaults.ts);
   E04 should write a fuller one (capacity check, deployment-list check,
   env-var presence, dataset version present, etc.).

2. **Cloud-resource setup belongs in the brief, pre-listed.** The harness
   blocks paid `az ... create` calls. The brief should pre-list the exact
   commands the *user* runs ahead of time (and ideally before the agent
   starts). E04's brief already does this for `gpt-5.5` quota request +
   deployment; future experiments should follow that template.

3. **Workflow output-port ‚Üî activity-return-field contract should have a
   static test.** The `cuResponse ‚Üí ocrResponse` typo in E03 was a silent
   failure: metrics looked fine, `benchmark_ocr_cache` stayed empty.
   A repo-wide test asserting *"for every activity in the registry, every
   workflow JSON's `outputs[].port` is present in the activity's
   return-type"* would have caught it pre-commit. Easy to add, prevents
   a class of bugs across all engines.

4. **The iteration kit copy is now the standard starting move.** E02 ‚Üí E03
   copy got us to 97.3% per-field accuracy on `synth-full (1)` in a
   single iteration. E04 ‚Üí copy E03's iteration kit. The pattern works
   because the SDPR-form quirks (column conventions, blank-vs-zero, etc.)
   are engine-agnostic; only field-name ‚Üí field-purpose mapping
   transfers. Future experiment briefs should explicitly say "step 0 is
   to `cp -r ../03-content-understanding/iteration ../<slug>/iteration`."

5. **Draft SUMMARY.md skeleton on day 1.** Sections that don't depend on
   benchmark numbers (provider doc, vocabulary mapping, parent-shared
   infra fixes, retrospective) can be drafted while the iteration kit is
   tuning and again while the canonical run is running. By the time
   metrics are in, the SUMMARY is 70% done. Saved an hour at the end of
   E03.

6. **Trust the latest official doc, not external AI on rapidly-evolving
   APIs.** During E03 setup we bounced between three sources (an external
   AI assistant working from stale Microsoft Learn pages, my prior
   knowledge, the actual API reference). For the next CU/Foundry-style
   engine, spend the first 5 minutes WebFetching the current Microsoft
   Learn / API reference page; treat anything else as hypothesis until
   verified.

7. **`trigger-experiment-benchmark.ts` + `poll-experiment-run.ts` are
   reusable.** They live at `apps/temporal/src/scripts/`, accept any slug
   matching `seed-experiment-{slug}-definition`, and pass
   `runtimeSettingsOverride` for per-sample timeout (default 3600 s).
   The bash equivalent in `scripts/run-experiment-benchmarks.sh` doesn't
   accept the override, so prefer the TS wrapper for any
   generative-engine experiment. E04 should reuse them as-is.

8. **`TEST_API_KEY` env-loading is a three-file dance.** It actually
   lives in `apps/backend-services/.env` (repo-local), not the override
   file as the shared rules implied. Both the trigger and poll TS scripts
   load all three files in order ‚Äî E04 reusing them won't hit this. Only
   relevant if you write a new TS script that needs the key.

9. **There may be silent post-success failures.** Like E03's empty-cache
   bug. Add explicit "cache row count" assertion to the canonical run
   validation: after the run completes, query `benchmark_ocr_cache` and
   refuse to mark the run "canonical" if `total_samples != cache row
   count`. Worth folding into `poll-experiment-run.ts`.

10. **Pre-existing `graph-workflow.test.ts` failure** (29/29 fail on the
    parent-tip) blocks confidence in "all tests pass." Worth fixing on
    the chained stack so E04+E05 don't have to filter it out manually.
    Not E03's scope to fix; flagged for the cross-experiment cleanup
    after E05.

11. **Capacity bumps are cheap and reversible.** Bumping `gpt-5.2`
    capacity from 10 ‚Üí 100 cut benchmark wallclock from 40 min (with 31
    timeouts) to 6:44 (zero timeouts). Capacity is a throughput cap, not
    a reservation ‚Äî billing is per-token-used regardless. Future
    experiments should bump capacity to whatever the regional quota
    allows as a pre-flight step.

### Implications for E04 (VLM-direct)

E04's brief should bake in the lessons above. Specifically:

- **Reuse the gpt-5.2 deployment** ‚Äî E03's deployment + capacity-100 setup is reusable for E04's direct chat-completions calls. No new model deployment needed; same regional quota applies.
- **No analyzer plumbing** ‚Äî E04 is a direct VLM call (chat-completions with a vision input + structured-output JSON Schema). Skip CU's analyzer layer entirely. Iteration-kit prompts (`prompt.md`) port directly; per-field descriptions become JSON Schema `description` properties on each field.
- **Numeric-nullable handling** ‚Äî E04 can use OpenAI structured outputs' real `["number","null"]` union, unlike CU which had to encode nullability in the description. Cleaner.
- **Latency** ‚Äî E04 should be faster than E03 because CU's content-extraction layer is bypassed. Expect ~15-25 s per sample at capacity 100, vs E03's ~22 s. Same `runtimeSettingsOverride: { timeout: 3600 }` pre-flight applies.
- **Confidence semantics** ‚Äî chat-completions does not return a per-field confidence directly. E04 will need a different gating strategy for `ocr.checkConfidence` (e.g. always-high default, or token-logprob-based confidence if exposed).
- **Pre-flight: confirm structured-output mode is on.** OpenAI's chat-completions `response_format: { type: "json_schema", strict: true }` is the toggle. Without `strict: true` the model may return free-form JSON that doesn't match the schema. Mirror E02's check-strict-flag-immediately rule.

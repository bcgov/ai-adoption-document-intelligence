Authentication & Authorization Architecture

This page documents the authentication and authorization system used by the Document Intelligence Platform. The implementation follows the OAuth 2.0 Authorization Code Flow with PKCE, using Keycloak as the identity provider and a confidential NestJS backend as the OAuth client.

Dual Authentication
The platform supports two authentication methods: Bearer Tokens (Keycloak SSO) for user-facing applications and API Keys for machine-to-machine integrations. Both are enforced globally via NestJS guards.

Architecture Overview

The authentication system is split across the backend (NestJS) and frontend (React SPA), with Keycloak handling identity management.

Backend (Confidential Client)

  • Holds the client_secret — never exposed to the browser
  • Uses openid-client (v6.x) for OIDC discovery, code exchange, and token refresh
  • Uses passport-jwt + jwks-rsa for bearer token validation
  • PKCE (code_verifier / code_challenge) stored server-side
  • Short-lived in-memory session store for passing tokens to the SPA

Frontend (Stateless SPA)

  • No OIDC client library — relies entirely on the backend
  • Stores provider-issued tokens in localStorage
  • Proactive refresh timer at 75% of token lifetime
  • Axios 401 interceptor with single-flight refresh
  • Visibility change listener for tab-return refresh

OAuth 2.0 Login Flow

The login process uses the Authorization Code Flow with PKCE and a confidential client. The backend orchestrates all interactions with Keycloak — the browser never sees the client secret, authorization code exchange, or PKCE verifier.

Step-by-Step

StepActorAction
1 User Clicks "Login" — browser navigates to GET /api/auth/login
2 Backend Generates PKCE pair (code_verifier + code_challenge), random state, and nonce via openid-client
3 Backend Stores PKCE state in AuthSessionStore (in-memory, 5-minute TTL)
4 Backend Redirects browser to Keycloak authorization endpoint with code_challenge, state, nonce, and scope=openid profile email
5 User Authenticates at Keycloak (e.g., IDIR credentials)
6 Keycloak Redirects to GET /api/auth/callback?code=...&state=...
7 Backend Validates state, retrieves stored code_verifier and nonce
8 Backend Exchanges authorization code for tokens via openid-client.authorizationCodeGrant() — includes code_verifier and client_secret
9 Backend The openid-client library validates the ID token (signature via JWKS, nonce, issuer, audience)
10 Backend Stores tokens in AuthSessionStore with a UUID key (60-second TTL, one-time consumption)
11 Backend Redirects browser to FRONTEND_URL?auth_result=UUID
12 SPA Detects ?auth_result= in URL, fetches tokens via GET /api/auth/result?result=UUID
13 SPA Stores tokens in localStorage, decodes ID token for profile, schedules refresh timer
One-Time Token Consumption. The auth_result UUID can only be redeemed once. After consumption, the entry is deleted from memory. This prevents replay attacks via browser history or shared links.

Token Refresh Strategy

The platform uses a three-layer approach to keep sessions alive without requiring the user to re-authenticate.

1. Proactive Timer

A setTimeout fires at 75% of the access token's lifetime (minimum 10 seconds before expiry). This refreshes the token before any API call can fail.

2. Visibility Listener

When the user returns to the tab (visibilitychange event), tokens are checked. If less than 60 seconds remain, an immediate refresh is triggered.

3. Axios 401 Interceptor

If a request returns 401 Unauthorized, the interceptor attempts a single-flight refresh, retries the original request, and only logs the user out if refresh fails.

Refresh Endpoint

POST /api/auth/refresh
Content-Type: application/json

{
  "refresh_token": "eyJhbGciOi..."
}

Returns updated access_token, refresh_token, id_token, and expires_in. The backend proxies this to Keycloak via openid-client.refreshTokenGrant().

Bearer Token Validation

All API requests (except routes marked @Public()) are validated by a global Passport JWT guard using Keycloak's JWKS endpoint.

Guard Execution Order

Three guards are registered globally via NestJS APP_GUARD. They execute in this order on every request:

OrderGuardResponsibility
1 JwtAuthGuard Extends Passport AuthGuard('jwt'). Skips @Public() routes. Defers to API key guard if @ApiKeyAuth() and x-api-key header present. Otherwise validates Bearer token via KeycloakJwtStrategy.
2 ApiKeyAuthGuard Validates X-API-Key header on @ApiKeyAuth()-decorated routes. Skips if user already authenticated via JWT.
3 RolesGuard Checks @Roles() decorator metadata against request.user.roles. Requires at least one matching role.

Keycloak JWT Strategy

The KeycloakJwtStrategy (Passport strategy) handles token validation:

  • Key Provider: jwks-rsa with caching (24h) and rate limiting (5 req/min) — fetches Keycloak's RS256 public keys from /protocol/openid-connect/certs
  • Token Extraction: Authorization: Bearer {token} header
  • Validation: RS256 signature, issuer, audience, expiration
  • Role Normalization: Collects roles from three Keycloak JWT locations:
    • payload.roles[] — top-level claim
    • payload.realm_access.roles[] — realm-level roles
    • payload.resource_access.<client_id>.roles[] — client-specific roles

API Key Authentication

For machine-to-machine (M2M) integrations, routes decorated with @ApiKeyAuth() accept an API key via the X-API-Key header as an alternative to Bearer tokens.

GET /api/documents
X-API-Key: your-api-key-here

How It Works

  • API keys are generated and managed through the platform Settings page
  • Each user can have one active key at a time
  • Keys are validated by the ApiKeyAuthGuard via ApiKeyService
  • On validation, request.user is populated with the key owner's identity

When to Use

  • Automated pipelines and CI/CD
  • Service-to-service integrations
  • Scripts and CLI tools
  • Any headless (non-browser) consumer

Authorization (RBAC)

Role-Based Access Control is enforced via the @Roles() decorator and the global RolesGuard.

How Roles Are Applied

// Controller example
@Controller('admin')
export class AdminController {
  @Get('users')
  @Roles('admin', 'user-manager')  // Requires at least one
  async listUsers() { ... }
}
  • Roles in the decorator use OR logic — the user needs at least one of the listed roles
  • If no @Roles() decorator is present, any authenticated user can access the route
  • Roles are sourced from the validated JWT (Keycloak realm and client roles, normalized by KeycloakJwtStrategy)

Frontend Auth Architecture

The React SPA uses a single AuthContext provider that encapsulates all authentication logic. No OIDC client library is used — the frontend relies entirely on the backend for OAuth interactions.

API Service (Axios)

The apiService wraps Axios with authentication-aware interceptors:

  • Request interceptor: Attaches Authorization: Bearer {token} to all outgoing requests
  • Response interceptor: On 401, triggers a single-flight refresh (shared promise to avoid concurrent refreshes), retries the original request with new token, logs out if refresh fails

Security Mechanisms

PKCE (Proof Key for Code Exchange)

The backend generates a code_verifier (random string) and sends its SHA-256 hash (code_challenge) to Keycloak. On callback, the original code_verifier is sent with the code exchange — Keycloak verifies the hash matches, preventing authorization code interception attacks.

Nonce Validation

A random nonce is included in the authorization request and embedded in the ID token by Keycloak. The openid-client library validates that the ID token nonce matches the expected value, ensuring the token was issued for this specific request.

JWKS Signature Verification

All bearer tokens are verified using Keycloak's RS256 public keys fetched from the JWKS endpoint. Keys are cached for 24 hours and rate-limited to 5 requests per minute via jwks-rsa.

Short-Lived Session Store

Auth results and PKCE state are stored in-memory with a short TTL (60 seconds for tokens, 5 minutes for PKCE). Entries are consumed once and automatically cleaned up by a background sweeper.

Critique & Improvement Recommendations

The current implementation is functional and follows industry-standard patterns. The following areas have been identified as opportunities for further hardening, scalability, and operational improvements.

Priority Items. The first three items below affect production scalability and security posture. The remaining are improvements for maintainability and operational excellence.

1. In-Memory Session Store Is Not Horizontally Scalable

Issue: AuthSessionStore uses a JavaScript Map for both PKCE state and token results. In a multi-instance (horizontally scaled) deployment behind a load balancer, the callback request may hit a different instance than the one that created the PKCE state — causing authentication failures.

Recommendation: Replace with Redis (or another shared store) for PKCE state and auth results. Use a NestJS CacheModule with a Redis driver, or implement a simple Redis-backed store with the same save()/consume() interface. Set TTLs natively in Redis to eliminate the manual sweeper.

Workaround (current): Use sticky sessions (session affinity) at the load balancer level, but this reduces the benefits of horizontal scaling.

2. Token Storage in localStorage Is Vulnerable to XSS

Issue: Storing access_token and refresh_token in localStorage means any JavaScript running on the page (including from XSS vulnerabilities or compromised third-party scripts) can read and exfiltrate the tokens.

Recommendation: Migrate to HttpOnly, Secure, SameSite=Strict cookies for token storage. The backend would set tokens as cookies in the /api/auth/result response, and the browser would automatically attach them to all same-origin API requests. This eliminates JavaScript access to tokens entirely.

Trade-off: Cookie-based storage requires CSRF protection (e.g., double-submit cookie pattern or SameSite attribute), but significantly reduces the XSS attack surface. Consider using a BFF (Backend-For-Frontend) pattern where the backend manages all token lifecycle and the SPA never sees raw tokens.

3. No Token Revocation Check on the Backend

Issue: Bearer token validation only checks the JWT signature, issuer, audience, and expiration. If a token is revoked in Keycloak (e.g., user account disabled, admin-initiated session termination), the token remains valid until it naturally expires. This is inherent to stateless JWT validation.

Recommendation: Implement token introspection for sensitive operations. Use Keycloak's /protocol/openid-connect/token/introspect endpoint to check if a token is still active. This can be applied selectively (e.g., for admin operations, write operations) to avoid the performance overhead on every request. Alternatively, use short-lived access tokens (e.g., 5 minutes) combined with refresh token rotation to reduce the window of exposure.

4. Duplicate Decorator Definitions

Issue: The API_KEY_AUTH_KEY constant and @ApiKeyAuth() decorator are defined in two places: auth/api-key-auth.decorator.ts and decorators/custom-auth-decorators.ts. The guards import from the decorators/ version, making the auth/ version effectively dead code. This creates confusion and risks bugs if the wrong import is used.

Recommendation: Remove the duplicate definition. Consolidate into a single canonical location (either auth/ or decorators/) and update all imports.

5. No Rate Limiting on Auth Endpoints

Issue: The /api/auth/refresh and /api/auth/result endpoints have no rate limiting. While Keycloak itself provides some brute-force protection for the login flow, the backend refresh and result endpoints are direct targets for abuse.

Recommendation: Add @nestjs/throttler with appropriate rate limits. For example, limit /api/auth/refresh to 10 requests per minute per IP and /api/auth/result to 5 requests per minute per IP.

6. No Structured Logging for Authentication Events

Issue: Authentication events (successful login, failed refresh, token validation failure, logout) are logged using the default NestJS logger without structured metadata. This makes it difficult to audit authentication activity, detect anomalies, or trace issues in production.

Recommendation: Emit structured JSON log entries for all auth events with consistent fields: event_type, user_sub, client_ip, timestamp, result (success/failure), and reason (for failures). Integrate with a centralized logging system (e.g., ELK, Splunk) for monitoring and alerting.

7. User Type Uses Loose Index Signature

Issue: The User type in auth/types.ts includes [key: string]: unknown, allowing arbitrary JWT claims to pass through. While pragmatic, this weakens type safety downstream — consumers of request.user may access properties that don't exist without TypeScript catching the error.

Recommendation: Define explicit optional fields for known Keycloak claims (e.g., preferred_username, given_name, family_name, realm_access, resource_access) and remove the index signature. If pass-through of unknown claims is truly needed, consider using a separate rawClaims field.

8. No Automated Auth Integration Tests

Issue: While unit tests exist for individual components, there are no end-to-end integration tests that exercise the full OAuth flow (login → callback → result → API call → refresh → logout). Auth regressions may only be caught in manual testing or production.

Recommendation: Create integration tests using a Keycloak test container (e.g., testcontainers with the Keycloak Docker image). Test the full flow including: successful login, expired token refresh, revoked token handling, rate limit enforcement, and role-based access denial.

Benchmarking System Guide

The benchmarking system lets you systematically evaluate your document processing workflows by running them against curated datasets with ground truth, then measuring accuracy, detecting regressions, and comparing results across workflow versions.

Prerequisites
The benchmarking system requires MinIO (S3-compatible object storage), MLflow (experiment tracking), and Temporal (workflow orchestration) to be running. See the Benchmarking Technical Reference for setup instructions.

Overview

The benchmarking system is designed to answer one question: "How well does my workflow perform, and has it gotten worse?" It is workflow-agnostic — it can evaluate any graph workflow configuration, not just OCR. The system wraps your existing workflow execution with dataset management, automated evaluation, metric tracking, and regression detection.

Datasets

Versioned collections of test documents with ground truth annotations, stored in Git and DVC. Upload documents and ground truth through the UI; the backend handles all version control operations automatically.

Evaluation

Pluggable evaluators compare workflow outputs against ground truth. Schema-aware evaluators provide per-field precision/recall; black-box evaluators do JSON deep-equal comparison.

Tracking

Every run is logged to MLflow with parameters, metrics, tags, and artifacts. Set baselines, configure thresholds, and detect regressions automatically.

Core Concepts

Projects, Definitions, and Runs

The benchmarking system has a three-level hierarchy:

Project (organizational container — maps to an MLflow Experiment)
  └── Definition (the benchmark recipe — pins dataset, workflow, evaluator)
        └── Run (a single execution of that recipe)

Project

A logical grouping for related benchmarking work. When you create a project, an MLflow Experiment is automatically created to track all runs within it.

Example: "Invoice Extraction Quality Q1 2026"

Definition

A precise, reproducible specification of what to benchmark. Pins together a dataset version, split, workflow, evaluator, runtime settings, and artifact storage policy.

Run

A single execution of a definition. Produces aggregated metrics, per-sample results, and artifacts. Maps to an MLflow Run for detailed inspection.

Immutability Model

A definition starts as mutable. You can edit it freely before running it. The moment you trigger the first run, the definition becomes immutable. This ensures that every run against that definition used identical settings, making results directly comparable.

Editing an immutable definition
If you need to change something on an immutable definition, the system creates a new definition with an incremented revision number. The original and all its runs remain untouched.

Why Multiple Runs?

A definition pins the workflow configuration (the DAG shape), but not the implementation. Several things can change between runs:

Source of VariationExample
Activity code updates You deploy a new version of an extraction activity with a better prompt or algorithm. The DAG shape is the same, but the code running inside has changed.
External model/API changes A workflow node calls an external LLM or OCR service whose model has been updated.
Worker version You rebuild and deploy new worker Docker images. Each run records the workerGitSha and workerImageDigest so you can trace which code was running.
Non-deterministic behavior LLM responses with temperature > 0, race conditions in parallel processing, or external services returning different results.

Step 1: Prepare a Dataset

A dataset is a versioned collection of test documents with their expected outputs (ground truth). The platform manages dataset storage through Git + DVC backed by MinIO — you interact only through the UI and API.

Create a Dataset

Navigate to Benchmarking → Datasets in the sidebar and click Create Dataset. Provide:

FieldDescription
Name Human-readable name (e.g., "Income Declaration Forms")
Description Optional description of the dataset's contents and purpose
Repository URL Path to the dataset Git repository (must be unique). Supports local paths (~/Github/datasets-repo), file:// URLs, and remote HTTPS/SSH URLs.
Metadata Optional key-value metadata pairs — add fields dynamically using the Add Metadata button (e.g., domain, document type, language, source)

The backend automatically initializes a Git repository with DVC configured to use MinIO as the remote storage backend.

Upload Files

On the dataset detail page, click New Version to create a draft version — this automatically opens the file upload dialog. You can also upload additional files to an existing draft version using the Upload Files action in the version's menu. Upload up to 100 files at a time (max 100 MB each). The upload dialog supports drag-and-drop and lets you remove individual files before uploading.

Organize your files into two categories:

Input Documents

The files your workflow will process — PDFs, images, scanned forms. Place them in an inputs/ directory within your upload.

Ground Truth

The expected outputs — JSON files containing the correct extraction results. Place them in a ground-truth/ directory. Supports JSON, JSONL, CSV, or any file format your evaluator can parse.

Ground Truth Format

Ground truth files are flexible — the evaluator is responsible for parsing them. A typical key-value extraction ground truth JSON looks like:

{
  "name": "Edward Shaw",
  "sin": "104125381",
  "date": "2013-09-21",
  "income1": "999.91",
  "checkbox_need_assistance_no": true,
  "explain_changes": "Moving to a new address",
  "signature": "Edward Shaw"
}

Dataset Manifest

When you create a version, the system generates a manifest — a JSON file that maps each sample to its input files, ground truth files, and metadata:

{
  "schemaVersion": "1.0",
  "samples": [
    {
      "id": "sample-001",
      "inputs": [
        { "path": "inputs/form_image_0.jpg", "mimeType": "image/jpeg" }
      ],
      "groundTruth": [
        { "path": "ground-truth/form_data_0.json", "format": "json" }
      ],
      "metadata": {
        "docType": "income-declaration",
        "pageCount": 1,
        "language": "en"
      }
    }
  ]
}

Version Lifecycle

When you click New Version, the backend automatically:

  1. Runs dvc add on the uploaded files to track them
  2. Generates the manifest file
  3. Commits DVC metadata to Git
  4. Pushes large files to MinIO via dvc push
  5. Creates a DatasetVersion record with the Git commit SHA

Versions follow a three-state lifecycle:

Draft  ──▶  Published  ──▶  Archived
StatusDescriptionAvailable Actions
Draft Initial state. Files can still be uploaded, samples can be deleted. Upload Files, Validate, Publish, Delete Version
Published Available for use in benchmark definitions. No further uploads. Validate, Archive
Archived No longer available for new definitions. Existing definitions referencing it continue to work. Validate
Deletion restrictions
Published and archived versions cannot be deleted if they are referenced by any benchmark definition. Draft versions can be deleted freely.

Validate a Dataset Version

Before publishing, you should validate your dataset to catch common issues. Click Validate in the version's action menu (the three-dot icon) — this is available for versions in any status.

The validation report opens in a modal and checks for:

Check CategoryWhat It Detects
Schema Violations Ground truth files that don't match the expected structure
Missing Ground Truth Input documents that have no corresponding ground truth file
Duplicates Duplicate samples or files within the dataset
File Corruption Files that cannot be read or are malformed

Each issue is displayed with its sample ID, severity (error or warning), a message, and the affected file path. For large datasets, you can optionally specify a sample size to validate a random subset instead of the entire dataset.

Tip
Run validation after uploading files and before publishing. The report clearly indicates whether the dataset passed or failed with an overall status badge and a summary of issue counts.

View Ground Truth

On the Sample Preview tab, click any sample's ground truth link to open the Ground Truth Viewer — a modal that displays the JSON contents in a formatted, scrollable view. This is useful for spot-checking annotation quality before publishing.

Delete Samples

While a version is in draft status, you can delete individual samples from the Sample Preview tab. This is useful for removing incorrectly uploaded or duplicated files before publishing.

Splits

A split defines a subset of a dataset version used for evaluation. Manage splits from the Splits tab on the dataset detail page (select a version first).

Create a Split

Click Create Split and provide:

FieldDescription
Name Alphanumeric characters, hyphens, underscores, and dots only
Type Select from: Train, Validation, Test, or Golden Regression
Samples Multi-select picker listing all samples in the version

Split Types

Split TypeUse Case
trainTraining subset (for reference only; not used by evaluators)
valValidation/tuning subset
testStandard evaluation subset
goldenRegression test set — can be frozen to lock its sample list permanently

Edit and Freeze Splits

While a split is unfrozen, you can edit it to update the sample selection. The splits table shows each split's name, type, sample count, frozen/editable status, and creation date.

Golden splits can be frozen to make them permanently immutable — this ensures your regression tests always use the exact same documents. Freezing is irreversible; a confirmation dialog is shown before proceeding.

Delete a Dataset

You can delete an entire dataset from the dataset detail page. This is a cascading operation — all versions, splits, samples, and uploaded files are permanently removed.

Step 2: Create a Project

Navigate to Benchmarking → Projects and click Create Project. Provide a name and optional description.

Creating a project also creates a corresponding MLflow Experiment automatically. All runs within the project are tracked under this experiment in MLflow.

Step 3: Create a Benchmark Definition

Within a project, click Create Definition. A definition specifies exactly what to benchmark:

SettingDescription
Dataset Version Select a published dataset version
Split Choose which subset of the dataset to evaluate
Workflow Select a graph workflow to evaluate — the system records the workflow config hash at creation time
Evaluator Type Choose how results are scored (see Evaluators section below)
Evaluator Config Evaluator-specific settings (matching rules, thresholds, field types)
Runtime Settings Concurrency limits, timeouts, and queue selection
Artifact Policy What outputs to store after the run completes

Evaluator Types

The system includes three built-in evaluators:

Schema-Aware Evaluator

Compares structured field outputs against typed ground truth at the field level. Produces precision, recall, and F1 score per field. Supports configurable matching rules:

  • Exact match — String equality comparison
  • Fuzzy match — Levenshtein/Jaro-Winkler distance with configurable similarity threshold
  • Numeric tolerance — Absolute or relative numeric comparison (e.g., ±0.01)
  • Date normalization — Parses and compares dates across multiple formats
  • Boolean match — Checkbox/boolean accuracy

Best for: Structured data extraction workflows where you know the expected output fields.

Black-Box Evaluator

JSON deep-equal comparison with diff output. Falls back to raw byte comparison for non-JSON files. Produces exact_match and field_overlap metrics along with detailed diff artifacts.

Best for: Comparing workflow outputs where you don't need field-level granularity, or when the output schema isn't formalized.

Field Accuracy Evaluator

Lightweight evaluator that computes field_accuracy, character_accuracy, and word_accuracy metrics. Useful for quick character-level and word-level comparison.

Evaluator Configuration Example

For the schema-aware evaluator, you can specify per-field matching rules:

{
  "matchingRules": {
    "name": { "type": "fuzzy", "threshold": 0.85 },
    "income1": { "type": "numeric", "tolerance": 0.01 },
    "date": { "type": "date", "formats": ["YYYY-MM-DD", "MM/DD/YYYY"] },
    "checkbox_need_assistance_no": { "type": "boolean" }
  },
  "defaultRule": { "type": "exact" }
}

Runtime Settings

{
  "maxParallelDocuments": 10,
  "perDocumentTimeout": 300000,
  "perRunTimeout": 86400000,
  "useProductionQueue": false
}
SettingDescription
maxParallelDocuments Maximum number of documents processed in parallel within a single run (default: 10)
perDocumentTimeout Timeout in milliseconds per document (default: 5 minutes)
perRunTimeout Overall timeout for the entire benchmark run (default: 24 hours)
useProductionQueue If true, use the production task queue instead of the isolated benchmark queue. Requires deliberate opt-in.

Artifact Policies

PolicyBehavior
full Store all outputs for every document
failures_only Only store outputs for documents that had errors or low scores
sampled Store a configurable percentage of outputs

Step 4: Start a Benchmark Run

From a definition's detail page, click Start Run. This triggers the following sequence:

  1. A BenchmarkRun record is created with status pending
  2. The definition is locked to immutable (on first run)
  3. An MLflow Run is created in the project's MLflow Experiment
  4. A benchmarkRunWorkflow Temporal workflow is started on the benchmark-processing task queue

What Happens During a Run

The Temporal workflow executes a 7-phase pipeline:

Phase 1: Materialize Dataset

Clones the dataset Git repository at the pinned version, runs dvc pull to fetch the actual data files from MinIO. Results are cached on the worker to avoid redundant downloads.

Phase 2: Load Manifest & Filter by Split

Reads the dataset manifest and filters samples to only include those in the selected split.

Phase 3: Fan-Out Execution

Processes samples in configurable batches. Each sample is executed through the selected workflow by spawning a child graphWorkflow — the same workflow function that production documents use. This ensures benchmarks test the actual execution path.

Phase 4: Evaluate Each Sample

Runs the configured evaluator on each (prediction, ground truth) pair, producing per-sample metrics and diagnostics.

Phase 5: Aggregate Metrics

Computes dataset-level statistics: mean, median, standard deviation, percentiles (p5, p25, p75, p95), min, and max for each metric. Also performs failure analysis — identifies worst-performing samples, per-field error breakdowns, and error clustering.

Phase 6: Log to MLflow

Logs all parameters, metrics, tags, and artifacts to the corresponding MLflow Run. Artifact upload follows the definition's artifact policy.

Phase 7: Baseline Comparison & Cleanup

If a baseline run exists for this definition, computes metric deltas and checks thresholds. Cleans up temporary materialized files on the worker.

Monitoring Progress

While a run is in progress, the run detail page auto-polls for updates — you do not need to refresh the page. It shows real-time progress including the current phase, document counts (processed/total), and percentage completion.

Available actions during and after a run:

ActionWhen AvailableDescription
Cancel Pending or Running Stop the benchmark immediately
Re-run Completed or Failed Start a new run from the same definition
Promote to Baseline Completed (not already baseline) Set this run as the reference for regression detection
Edit Thresholds Baseline run Update the regression thresholds on the current baseline
View Regression Report Completed with baseline comparison Navigate to the full regression analysis page

The run detail page also displays links to MLflow (for metric exploration) and Temporal UI (for workflow execution details), along with worker metadata such as the Git SHA and Docker image digest.

Step 5: View Results

After a run completes, the Run Detail Page (/benchmarking/projects/:id/runs/:runId) shows:

Metrics Summary

  • Aggregated metrics (mean, median, std dev, percentiles)
  • Per-field accuracy breakdown (for schema-aware evaluator)
  • Pass/fail counts
  • Timing information (duration, per-document average)

Baseline Comparison

  • Metric deltas vs. baseline
  • Threshold pass/fail status
  • Severity classification (Warning / Critical)
  • Regression badges on metrics that dropped

Drill-Down Analysis

The run detail page includes a Drill-Down Summary at the bottom, showing the top worst-performing samples, per-field error breakdowns, and error cluster tags. Click View All Samples to open the full drill-down page at /benchmarking/projects/:id/runs/:runId/drill-down.

The drill-down page provides:

  • Dynamic metadata filters — Filter panel with dropdowns for each metadata dimension (document type, language, etc.), with an active filter count badge and a clear-all button
  • Paginated results table — Sample ID, metadata dimensions, metrics badges, and a click action to view details
  • Sample detail drawer — Opens from the right side when clicking a sample. Shows the sample's metadata, individual metrics, ground truth JSON, prediction JSON, evaluation details, and diagnostics

Per-Sample Results

The Samples tab shows paginated per-sample results with:

  • Individual metrics per document
  • Pass/fail status
  • Filterable by metadata dimensions
  • Ground truth vs. prediction comparison

Artifacts

The Artifacts section lists all output files produced during the run, filterable by type:

Artifact TypeDescription
per_doc_outputWorkflow output for each document
intermediate_node_outputOutput from individual graph nodes
diff_reportComparison diffs between prediction and ground truth
evaluation_reportEvaluator output files
error_logError details for failed samples

Click any artifact row to open the Artifact Viewer, a side drawer that shows:

  • Metadata — Type, MIME type, sample ID, node ID, and file size
  • Content preview — Images are rendered inline, JSON is formatted, and text/CSV/log files are shown in a monospace viewer
  • Download — Save the artifact file locally
  • Open in MLflow — Link directly to the artifact in the MLflow UI

MLflow Integration

Each run links directly to MLflow for deeper inspection. In MLflow you can:

  • Compare parameters and metrics across multiple runs
  • Browse and download artifacts
  • Use MLflow's built-in charting and comparison tools
  • Search runs by tags, parameters, or metric values

Step 6: Establish a Baseline

Once you have a run with satisfactory results, promote it to baseline. This sets the reference point for regression detection.

On the run detail page, click Promote to Baseline and configure thresholds:

{
  "thresholds": {
    "f1_score": { "type": "absolute", "value": 0.90 },
    "precision": { "type": "relative", "value": 0.05 }
  }
}

Threshold Types

TypeMeaningExample
absolute Metric must be at or above this value F1 ≥ 0.90
relative Metric must not drop by more than this fraction relative to the baseline Precision must not decrease by more than 5%
One baseline per definition. Only one run can be the baseline for a given definition at a time. Promoting a new baseline automatically demotes the previous one. All baseline changes are recorded in the audit log.

After a baseline is established, every subsequent completed run is automatically compared against it. Regressions are flagged with severity levels (Warning / Critical) on the run list and detail pages.

When viewing a completed run that has no baseline comparison, the run detail page shows a suggestion to promote it to baseline. This makes it easy to establish your first baseline.

Baseline Change History

All baseline promotions are recorded in an audit log. The definition detail view includes a Baseline Change History table showing when each baseline was promoted, which run was promoted, and by whom. This provides full traceability for your quality gates.

Step 7: Schedule Recurring Runs

Set up automated benchmark runs on a cron schedule for continuous regression monitoring.

From the definition detail page, click Configure Schedule:

{
  "enabled": true,
  "cron": "0 2 * * *"
}

This example runs the benchmark every night at 2:00 AM. Schedules are managed by Temporal's built-in scheduling system.

The schedule configuration panel shows the current status including the schedule ID, cron expression, next scheduled run time, and last run time, along with an Active/Paused badge.

Use nightly schedules with a baseline to automatically detect regressions after each deployment. Check the project page each morning for regression badges on the latest run.

Step 8: Compare Runs

Select 2–5 runs from the project page and click Compare to open a side-by-side comparison view at /benchmarking/projects/:id/compare?runs=id1,id2,...

The comparison page shows:

  • Run information — Status, definition, and start time for each run
  • Metrics comparison — All metrics side by side with delta and delta % (color-coded green/red)
  • Parameter differences — Highlighted with "Changed" badges when parameters differ between runs
  • Tags comparison — Side-by-side with "Changed" badges for differing values
  • Export CSV — Download the comparison table as a CSV file
  • Export JSON — Download the full comparison data as JSON

Step 9: Regression Reports

When a run has a baseline comparison, view the full regression report at /benchmarking/projects/:id/runs/:runId/regression

The report includes:

Metric-by-Metric Analysis

  • Current value vs. baseline value
  • Delta and delta % for each metric
  • Threshold configuration and pass/fail status
  • Severity classification (Warning / Critical)
  • Toggle to show only regressions
  • Click any metric row for a drill-down drawer with investigation suggestions

Historical Trend Chart

  • Interactive line chart showing metric values over time
  • Multi-select metrics to display
  • Date range presets: Last 10/20 runs, Last 30/90 days, All runs
  • Current run highlighted with a distinct marker
  • Threshold reference lines shown as dashed red lines
  • Clickable legend to toggle metric visibility

Common Scenarios

Evaluating a New Model Version

  1. You already have a project with a definition and a baseline run
  2. Deploy the new model version to your workers
  3. Re-run the same definition — the new run uses the updated model code
  4. Compare the new run against the baseline on the run detail page
  5. If metrics improved, promote the new run to the new baseline

Nightly Regression Monitoring

  1. Create a definition with your production workflow and a frozen golden dataset split
  2. Promote an initial run to baseline with thresholds
  3. Enable a nightly cron schedule on the definition
  4. Every morning, check the project page for regression badges on the latest run

A/B Testing Two Workflow Approaches

  1. Create two definitions in the same project — one for each workflow variant
  2. Run both against the same dataset version and split
  3. Use the comparison page to evaluate metrics side by side

Editing a Definition After It Has Been Run

  1. You realize you need to change the evaluator config
  2. Update the definition — the system creates a new definition (revision 2) since the original is immutable
  3. The original definition and all its runs remain intact
  4. Run the new definition to get results with the updated config

Benchmarking System — Technical Reference

This page covers the internal architecture, infrastructure setup, data model, and development configuration of the benchmarking system. For a usage-focused guide, see the Benchmarking Guide.

Architecture Overview

The benchmarking system is built on four infrastructure components that work together with the existing NestJS backend and Temporal workflow engine:

MinIO (S3-Compatible Storage)

Unified object storage for DVC-managed dataset files, MLflow artifacts, and benchmark run outputs. Provides S3-compatible APIs consumed by DVC, MLflow, and the application's MinioBlobStorageService.

Ports: 19000 (API), 19001 (Console)

MLflow (Experiment Tracking)

Tracks benchmark experiments, runs, parameters, metrics, and artifacts. Backend store uses PostgreSQL (separate mlflow database on the same instance). Artifact store uses MinIO.

Port: 5000

PostgreSQL (Metadata Store)

Existing PostgreSQL 15 instance extended with benchmark tables (via Prisma). Also hosts the MLflow backend store in a separate mlflow database with a dedicated role.

Port: 5432

Temporal (Workflow Orchestration)

Existing Temporal deployment. Benchmark runs execute as benchmarkRunWorkflow on a dedicated benchmark-processing task queue, isolated from production workloads.

Port: 7233 (gRPC), 8088 (UI)

System Flow

User creates Dataset
  → Backend inits Git repo + DVC → files stored in MinIO (s3://datasets)
       ↓
User creates BenchmarkProject
  → MLflow Experiment created automatically
       ↓
User creates BenchmarkDefinition
  → Pins dataset version + split + workflow + evaluator
       ↓
User starts BenchmarkRun
  → MLflow Run created → Temporal workflow launched
       ↓
Temporal benchmarkRunWorkflow (7 phases):
  1. materializeDataset   — git clone + dvc pull from MinIO
  2. loadManifest         — read manifest, filter by split
  3. fan-out execution    — each sample → child graphWorkflow
  4. evaluate             — evaluator scores each (prediction, ground truth)
  5. aggregate            — compute stats (mean, median, p5–p95, failure analysis)
  6. logToMlflow          — params, metrics, tags, artifacts
  7. compareBaseline      — threshold checks → update BenchmarkRun in Postgres
  8. cleanup              — remove temp files
       ↓
Results viewable in frontend + MLflow UI

Local Development Setup

Prerequisites

  • Docker & Docker Compose — Required for MinIO, MLflow, PostgreSQL, and Temporal
  • Node.js 18+ and npm
  • Git — Required on the backend for DVC operations (dataset repository management)
  • DVC — Data Version Control CLI, installed on the backend service. The Dockerfile installs it automatically; for local development, install via pip install dvc[s3]

1. Start Infrastructure Services

The MinIO and MLflow services are defined in apps/backend-services/docker-compose.yml alongside the existing PostgreSQL container:

cd apps/backend-services
docker compose up -d

This starts five services:

ServiceImagePortsPurpose
postgres postgres:15 5432 Main application DB + MLflow backend store (mlflow database)
minio minio/minio 19000, 19001 S3-compatible object storage for DVC, MLflow artifacts, benchmark outputs
minio-init minio/mc — Init container: creates required MinIO buckets
postgres-init postgres:15 — Init container: bootstraps mlflow database and role
mlflow ghcr.io/mlflow/mlflow:latest 5000 MLflow tracking server
VS Code tasks
The project includes VS Code tasks for health-checking all services: wait: minio ready, wait: mlflow ready, wait: backend postgres ready. The Dev: prerequisites compound task runs them all in sequence.

Add the following to your apps/backend-services/.env file:

3. Run Database Migrations

The benchmark tables are part of the shared Prisma schema. Run migrations as usual:

cd apps/backend-services
npm run db:generate
npx prisma migrate dev

4. Start the Temporal Worker

The benchmark workflow and activities are registered in the Temporal worker. Start it with:

cd apps/temporal
npm run dev

The worker polls both the production ocr-processing queue and the benchmark-processing queue.

Key Services

DatasetService (1906 lines)

The largest service — orchestrates all dataset operations:

  • createDataset() — Clones or creates Git repository, initializes DVC with MinIO remote, creates Postgres record. Supports tilde (~) expansion for portable paths.
  • uploadFiles() — Writes uploaded files to the dataset repo on disk
  • createVersion() — Clones repo → generates manifest → runs dvc add → commits to Git → creates DatasetVersion record
  • publishVersion() — Transitions version from draft to published
  • Split management — Create/manage splits with stratification rules and freeze support
  • Ground truth validation — Uses JSON Schema (Ajv) to validate against declared schema
DvcService (364 lines)

Wraps Git and DVC CLI operations. All DVC automation is transparent to the user — they never interact with Git or DVC directly.

  • cloneRepository() — Git clone with optional credentials injection
  • createNewRepository() — git init + initial commit
  • initRepository() — dvc init + configure MinIO S3 remote
  • configureRemote() — Set DVC remote endpoint, access keys, SSL settings for MinIO
  • addFiles() — dvc add to track files
  • commitChanges() — Git commit of DVC metadata
  • pushData() — dvc push to MinIO
  • pullData() — dvc pull from MinIO
  • checkoutRevision() — Checkout specific Git commit

Supports tilde expansion (~/path → /home/user/path) for repository URLs.

MLflowClientService (330 lines)

HTTP client wrapping the MLflow REST API (v2.0). All MLflow interactions go through this service.

  • createExperiment(name) → experiment_id
  • createRun(experimentId, runName) → run_id
  • logParams(runId, params) — Log key-value parameters
  • logMetrics(runId, metrics) — Log numeric metrics with timestamps
  • logTags(runId, tags) — Log string tags
  • updateRunStatus(runId, status) — RUNNING/FINISHED/FAILED/KILLED
  • logArtifact(runId, artifactPath) — Upload file artifact
  • getRunDetails(runId) — Fetch run info from MLflow
  • searchRuns(experimentId) — Query runs within an experiment

Configured via MLFLOW_TRACKING_URI (default: http://localhost:5000).

BenchmarkTemporalService (369 lines)

Temporal client wrapper for starting and managing benchmark workflows.

  • startBenchmarkRunWorkflow(runId, definition) — Starts benchmarkRunWorkflow on benchmark-processing task queue with 24h timeout
  • cancelBenchmarkRunWorkflow(workflowId) — Cancel via Temporal handle
  • getWorkflowStatus(workflowId) — Describe workflow state
  • createSchedule(workflowId, cron, ...) — Create Temporal Schedule for recurring runs
  • pauseSchedule(), resumeSchedule(), deleteSchedule(), getScheduleInfo()
BenchmarkRunService (809 lines)

Manages the run lifecycle:

  • startRun() — Validates definition → gets workerGitSha via git rev-parse HEAD → creates MLflow run → creates Postgres record → starts Temporal workflow → marks definition immutable → writes audit log
  • cancelRun() — Sends cancel signal to Temporal workflow, updates status
  • getDrillDown() — Returns worst-performing samples, field error breakdown, error clusters
  • getPerSampleResults() — Paginated per-sample results with metadata dimension filtering
  • promoteToBaseline() — Demotes previous baseline, sets thresholds, records audit log
BenchmarkArtifactService (230 lines)

Manages artifact storage in MinIO:

  • createArtifact(input) — Writes to MinIO with key format {runId}/{type}/{sampleId}-{nodeId}-{timestamp}.ext
  • listArtifacts(projectId, runId, type?) — List with optional type filter
  • getArtifactContent(projectId, runId, artifactId) — Stream from MinIO

Uses MinioBlobStorageService and bucket MINIO_ARTIFACT_BUCKET (default: benchmark-outputs).

Temporal Workflows & Activities

Benchmark Workflow

The central orchestrator is benchmarkRunWorkflow defined in apps/temporal/src/benchmark-workflow.ts (778 lines). It coordinates the 7-phase pipeline described above.

FeatureDetails
Task Queue benchmark-processing (isolated from production ocr-processing)
Timeout 24 hours (configurable)
Signal cancelBenchmarkSignal — for cancellation
Query getProgress — returns current phase, processed/total counts, percent complete

Activities

ActivityFileDescription
materializeDataset activities/benchmark-materialize.ts Clones dataset Git repo at pinned revision, runs dvc pull, caches results in /tmp/benchmark-cache/{datasetId}-{gitRevision}
benchmarkEvaluate activities/benchmark-evaluate.ts Routes evaluation to the appropriate evaluator from the registry, runs per-sample scoring
benchmarkAggregate activities/benchmark-evaluate.ts Computes aggregate metrics (mean, median, stdDev, percentiles) and failure analysis
benchmarkLogToMlflow activities/benchmark-logging.ts Logs params, metrics, tags, and artifacts to MLflow REST API
benchmarkCleanup activities/benchmark-logging.ts Removes temporary materialized files from worker filesystem
benchmarkUpdateRunStatus activities/benchmark-update-run.ts Updates BenchmarkRun record in Postgres with status, metrics, and timing
benchmarkCompareAgainstBaseline activities/benchmark-baseline-comparison.ts Finds baseline run, computes metric deltas, checks absolute/relative thresholds
Child workflow execution
During the fan-out phase, each sample is processed by spawning a child graphWorkflow — the same workflow function that production documents use. This is handled at the workflow level (not an activity) in activities/benchmark-execute.ts, which uses executeChild('graphWorkflow', ...) to run on the benchmark-processing queue.

Evaluator Implementations

The real evaluator logic runs in the Temporal worker (at apps/temporal/src/evaluators/), while the backend has lightweight stubs for registration purposes.

Schema-Aware Evaluator (497 lines)

Field-level comparison engine with configurable matching rules:

  • Exact match — Strict string equality
  • Fuzzy match — Levenshtein/Jaro-Winkler distance with configurable similarity threshold
  • Numeric tolerance — Absolute or relative numeric comparison
  • Date normalization — Parses dates across multiple formats and compares normalized values
  • Boolean match — Boolean/checkbox accuracy

Produces per-field precision, recall, and F1 score. Supports FieldMatchingRule config with fuzzy thresholds and numeric tolerances.

Black-Box Evaluator (376 lines)

JSON deep-equal comparison with detailed diff output:

  • Attempts JSON parse of both prediction and ground truth files
  • If both are valid JSON: performs deep-equal comparison, generates field-level diff
  • If either is not JSON: falls back to raw byte comparison
  • Produces exact_match (0 or 1) and field_overlap (percentage of matching fields) metrics
  • Generates diff artifacts showing differences between prediction and ground truth

Aggregation Engine

The statistical aggregation engine (apps/temporal/src/benchmark-aggregation.ts, 717 lines) computes:

  • Per-metric statistics: mean, median, standard deviation, p5, p25, p75, p95, min, max
  • Failure analysis: worst-performing samples by any metric, per-field error statistics, error clustering by type/pattern
  • Types: MetricStatistics, AggregatedMetrics, WorstSample, FieldErrorStats, ErrorCluster, FailureAnalysis

Blob Storage Layer

The benchmarking system uses MinioBlobStorageService for all object storage, alongside the existing Azure and local filesystem implementations:

apps/backend-services/src/blob-storage/
├── blob-storage.module.ts          # Provides all three implementations
├── blob-storage.interface.ts       # BlobStorageInterface (write/read/exists/delete)
├── blob-storage.service.ts         # Azure Blob Storage (production documents)
├── local-blob-storage.service.ts   # Local filesystem (development)
└── minio-blob-storage.service.ts   # S3-compatible MinIO (benchmarking)

MinioBlobStorageService implements BlobStorageInterface using @aws-sdk/client-s3 with forcePathStyle: true for MinIO compatibility. It handles:

  • write(key, data) — PutObject to MinIO
  • read(key) — GetObject from MinIO (returns Buffer)
  • exists(key) — HeadObject check
  • delete(key) — DeleteObject

Troubleshooting

MLflow connection refused

If MLflow is not reachable at http://localhost:5000:

  1. Check that the MLflow container is running: docker ps | grep mlflow
  2. Check MLflow logs: docker logs ai-doc-intelligence-mlflow
  3. Verify the PostgreSQL mlflow database exists: connect to Postgres and run \l
  4. Ensure MinIO is healthy (MLflow needs MinIO for artifact storage)
MinIO bucket not found

If you get "NoSuchBucket" errors:

  1. Check that the minio-init container ran successfully: docker logs ai-doc-intelligence-minio-init
  2. Manually create buckets via the MinIO Console at http://localhost:19001
  3. Or use the MinIO CLI: mc mb local/datasets local/mlflow-artifacts local/benchmark-outputs
DVC operations failing

Common DVC issues:

  • DVC not installed: Install with pip install dvc[s3]. The [s3] extra is required for MinIO/S3 support.
  • Remote not configured: DVC remote is configured automatically when creating a dataset. Check the repo's .dvc/config file.
  • MinIO connection issues: DVC uses MINIO_ENDPOINT, MINIO_ACCESS_KEY, and MINIO_SECRET_KEY env vars. Verify they match your MinIO instance.
  • SSL errors: For local MinIO without SSL, ensure DVC remote has endpointurl set to http:// (not https://) and use_ssl = false.
Dataset repository path not found

Tilde expansion issues:

  • Tilde (~) is expanded to the current user's home directory on the backend server, not the frontend user's machine
  • Ensure the expanded path exists and is accessible by the backend process
  • For Docker deployments, tilde expands to the container user's home directory
  • Prefer DEFAULT_DATASET_REPOSITORY_PATH env var for consistent configuration across environments
Benchmark workflow stuck in pending

If runs stay in pending status:

  1. Verify the Temporal worker is running and polling the benchmark-processing queue
  2. Check the Temporal UI at http://localhost:8088 for the workflow execution
  3. Ensure the BENCHMARK_TASK_QUEUE env var matches between the backend service and the Temporal worker
  4. Check Temporal worker logs for registration errors
